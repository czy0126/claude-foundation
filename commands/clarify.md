# /clarify — 提示词澄清助手

分析**我在本次对话中最近的一条任务请求**，按以下流程帮我改进。

## 流程

### 1. 诊断（5 维评分）

| 维度 | 评分 | 缺失了什么？ |
|------|------|-------------|
| 清晰度 | 1-5 | 目标能否一句话说清？ |
| 具体性 | 1-5 | 有文件路径、函数名、具体行为？ |
| 上下文 | 1-5 | 有技术栈、相关文件、项目约定？ |
| 完整性 | 1-5 | 有约束+验证+排除范围？ |
| 结构 | 1-5 | 分段/编号/用标题组织？ |
| 总分 | X/25 | |

### 2. 💡 建议卡

基于对任务的理解，先给 2-3 个可行方向（每个 ≤3 行），附推荐理由：
```
1. [方案A] — 适合[场景]，代价[代价]
2. [方案B] — 适合[场景]，代价[代价]
→ 推荐 [方案]，因为 [理由]
```
只有一个合理方案时直说，不硬凑。

### 3. 追问（按需）

如果评分最低的维度 ≤3 分，针对它们提 ≤3 个问题，每题带推荐选项：
```
> 关于[维度]：
> A) [选项] ← 推荐
> B) [选项]
```
所有维度 ≥4 分 → 跳过追问，直接重写。

### 4. 重写

输出重写后的高质量版本。原则：目标型而非指令型，补全上下文，标注验证方式。

### 5. 对比 + 判断权

```
【改进前】[原始]
【改进后】[重写]
【关键改进】+ 1-2 点

→ A) 通过  B) 调整细节  C) 重新来
```
连续 3 轮选 B/C → 不再追问，用当前版本执行，记录偏好到项目 CLAUDE.md。
