# 提示词工艺手册 — 从"说不清楚"到"精准驱动"

> 提示词书写是 LLM 编程最被低估的技能。研究显示：**提示词措辞差异可导致 20-50% 的性能波动**，而大部分用户将 AI 输出不佳归咎于"模型变笨了"而非自己没说清楚。
>
> 此手册提供可执行的提示词改进方法论。目标读者：**不擅长写 prompt 的使用者**。

---

## §零、5 分钟快速入门

**如果你只有 5 分钟**，按顺序做这三件事：

### 第一步：用三要素检查你的 prompt（1 分钟）

```
□ 上下文：给了文件路径吗？（@file:src/xxx.py）
□ 行动：说了要达成什么效果吗？（不是怎么做，是做什么）
□ 标准：写了怎么算完成吗？（测试命令？手动验证？）
```

三要素不缺 → 可以直接发送。缺了 → 补上再发。

### 第二步：如果还不行，加 Echo Prompt（1 分钟）

在 prompt 末尾加：
```
如果我的请求不够清晰，请先帮我重写为更好的提示词再回答。
```

### 第三步：用模板（3 分钟）

翻到第七章，复制匹配你场景的模板（功能开发/Bug 修复/审查/重构/调试/探索），填占位符。

> 💡 **最常见的错误**：写"怎么做"（指令型）而非"要什么结果"（目标型）。把"在 order.ts 第 15 行加 if"改成"确保 createOrder 拒绝非法金额"。

## 一、为什么大部分提示词会失败

### 失败提示词的 5 种症状

| 症状 | 表现 | 根因 |
|------|------|------|
| **泛泛而谈** | AI 输出笼统、无针对性 | 缺上下文（文件路径、技术栈、约束） |
| **过度自信** | 输出看起来对但实际错 | 缺验证标准，AI 不知道"对"是什么 |
| **格式漂移** | 输出格式与期望不符 | 未指定输出结构/格式 |
| **部分服从** | 只执行了一部分指令 | 长提示词中关键约束被稀释 |
| **幻觉补充** | AI 自己加了不该加的东西 | 未明确排除边界和禁止事项 |

### 核心洞察

> **指令型**prompt（"在 order.ts 第 15 行加 if 判断"）把 AI 的能力上限卡在使用者的思维水平。
> **目标型**prompt（"确保 createOrder 处理所有非法金额——负数、零、超大值、非数字输入"）让 AI 发挥完整的模式识别能力。
>
> 来源：Anthropic Claude Code 团队博文 [Lessons from building Claude Code](https://claude.com/blog/lessons-from-building-claude-code-prompt-caching-is-everything)

## 二、黄金法则：目标优于指令

这是 Claude Code 官方团队认定的 **#1 提示词原则**。

```
❌ 指令型 — "打开 order.ts，找到 createOrder 函数，
              在第 15 行加一个 if 判断 amount > 0"

✅ 目标型 — "确保 createOrder 能处理所有非法金额——
             负数、零、超大值、非数字输入——
             遵循项目的错误处理惯例"
```

**为什么目标型更好**：
- 指令型假设使用者知道所有需要改的地方，AI 只是打字机
- 目标型让 AI 做它擅长的——模式匹配、边界条件发现、跨文件一致性
- 实践中目标型产出质量显著优于指令型（Anthropic 团队社区实践报告）

### 目标型提示词模板

```
请 [达成什么效果]，要求：
- 输入：[从哪取数据/参数]
- 输出：[产出什么]
- 约束：[不能做什么、必须满足什么]
- 验证：[怎么证明完成了]
- 范围：[明确不处理什么]
```

## 三、提示词结构框架

### 3.1 基础三要素：上下文 → 行动 → 成功标准

| 要素 | 弱 | 强 |
|------|-----|-----|
| **上下文** | "在代码里" | "在 `src/api/users.ts` 第 34 行" |
| **行动** | "改进一下" | "抽取为纯函数，单一职责" |
| **成功标准** | （缺失） | "已有测试必须通过，新增 3 个边界用例" |

研究显示：**结构化提示词产出可用结果的概率高于模糊提示词**（参见 Prompt Architect 实证研究）。

### 3.2 CO-STAR 框架（推荐）

来自新加坡 GovTech 数据科学团队，被 Prompt Architect 采用：

| 字母 | 含义 | 填写内容 |
|------|------|---------|
| **C**ontext | 背景 | 项目技术栈、所在目录、相关文件 |
| **O**bjective | 目标 | 要达到什么具体效果 |
| **S**tyle | 风格 | 代码风格、命名约定、注释密度 |
| **T**one | 语气 | 正式/简洁/学术？ |
| **A**udience | 受众 | 写给谁看？高级工程师还是新手？ |
| **R**esponse | 输出格式 | Markdown/JSON/纯代码？结构要求？ |

### 3.3 CRISPE 框架

| 字母 | 含义 |
|------|------|
| **C**apacity & Role | 角色设定（"你是一个资深 Python 后端工程师"） |
| **R**equest | 核心请求（一句话） |
| **I**nsight | 背景洞察（"这个项目已经做过一轮重构，注意保持接口兼容"） |
| **S**tatement | 声明约束（"不要引入新依赖，不要改 public API"） |
| **P**ersonality | 输出风格（"输出简洁，不解释显而易见的决定"） |
| **E**xperiment | 实验/验证（"运行 pytest 确认通过"） |

### 3.4 何时用哪个框架

> **默认推荐：三要素（上下文-行动-标准）**。大多数日常任务用三要素就够了。CO-STAR 和 CRISPE 适合需要精确控制的高级场景。

| 场景 | 推荐框架 | 原因 |
|------|---------|------|
| **日常开发任务（默认）** | **三要素（上下文-行动-标准）** | 简单够用，学习成本最低 |
| 新功能开发 | 三要素 + 验证 | 对多数功能开发，三要素已足够 |
| 架构/设计任务 | CO-STAR 或 CRISPE | 需精确控制 Style/Tone/Audience |
| Bug 修复 | 三要素 + 复现步骤 | 上下文 = 错误日志 + 触发条件 |

## 四、常见提示词失败模式与修复

基于 mr-prompter 项目（31 种技术、13 种失败模式）的研究总结：

### 4.1 范围失败

| 失败模式 | 症状 | 修复 |
|---------|------|------|
| **F01 范围过宽** | "重构整个项目" → AI 动太多文件 | 限定到 1-3 个文件，< 100 行变更 |
| **F02 范围过窄** | "在 a.py 第 15 行加 x" → AI 成了打字机 | 描述目标和约束，让 AI 决定位置 |
| **F03 范围缺失** | 没说"不要改什么" → AI 动了不该动的 | 明确列出排除范围 |

### 4.2 格式失败

| 失败模式 | 症状 | 修复 |
|---------|------|------|
| **F04 无输出格式** | 你想要 JSON，AI 给了散文 | 写明 `输出必须是有效的 JSON，结构为 {...}` |
| **F05 格式模糊** | "输出表格" → 不知道是 markdown 还是 CSV | 指明具体格式 + 给示例 |
| **F06 缺少示例** | "写一个测试" → 不知道用什么框架/风格 | 给 1 个简短示例作为风格锚点 |

### 4.3 约束失败

| 失败模式 | 症状 | 修复 |
|---------|------|------|
| **F07 约束太松散** | "尽量保持简洁" → AI 不理解"尽量" | 说"不超过 200 行"而非"尽量简洁" |
| **F08 约束太刚性** | "必须用 asyncio" → 同步场景也用异步 | 说"推荐 asyncio，但如果是 CPU 密集用线程池" |
| **F09 约束冲突** | "最快速度"+"最完整功能" → AI 无所适从 | 明确优先级："正确性 > 性能 > 代码简洁性" |

### 4.4 上下文失败

| 失败模式 | 症状 | 修复 |
|---------|------|------|
| **F10 上下文不足** | 不知道项目用 React 还是 Vue | 在 CLAUDE.md 中记录技术栈 |
| **F11 上下文过载** | 贴了 500 行日志，关键信息被稀释 | 只说关键错误 + 摘录 20 行 |
| **F12 过时上下文** | 引用已删除的函数 | 先让 AI `grep` 确认存在，再基于它写代码 |
| **F13 隐式假设** | "像上次那样" — AI 没有上次的记忆 | 明确引用：用 @file 指向具体文件 |

## 五、Echo Prompt 技术

这是最**简单有效**的提示词改善方法——在提示词末尾加一句话，让 AI 帮你澄清意图。

### 基础版

```
[你的原始 prompt]

如果我的请求不够清晰，请先把它重写为一个更清晰、更有效的
提示词再回答——并简要说明你改了什么。
```

### 增强版（推荐用于 Claude Code）

```
[你的原始 prompt]

在开始之前，请先做以下检查：
1. 我的目标是否明确？（如果不明确，追问 1-2 个关键问题）
2. 我是否遗漏了必要的上下文？（如果遗漏，指出需要什么）
3. 我的约束是否清晰？（如果模糊，建议具体的约束措辞）

然后用你重写后的清晰版本来执行任务。
```

### 自动化版（写入 CLAUDE.md）

在 GLOBAL-CLAUDE.md 中添加以下规则后，Claude 会自动对模糊请求应用 Echo Prompt：

```
遇到以下任一情况时，先输出"需求澄清卡"再执行：
- 任务描述 < 20 字且不含文件路径
- 使用了"优化一下"、"改进"、"修一下"等无具体指向的动词
- 未指定验证标准
- 指令型而非目标型（告诉你怎么做而非要什么结果）
```

## 六、提示词质量自检清单

在发送 prompt 前，逐项打分（1-5 分）。总分 < 15 分 → 重写再发。

| 维度 | 1 分（差） | 3 分（可接受） | 5 分（优秀） |
|------|-----------|---------------|-------------|
| **清晰度** | "修一下 bug" | "修复登录页密码验证不生效的问题" | "修复 login.ts:34 处 bcrypt.compare 未 await 导致的验证失效" |
| **具体性** | 无文件路径 | 提到模块名 | `@file:src/auth/login.ts` 精确到行 |
| **上下文** | 无技术栈信息 | 提到语言/框架 | 引用 CLAUDE.md 中的约定 + 相关文件 |
| **完整性** | 只有"做什么" | 有"做什么"+输出要求 | 有目标+约束+验证+排除范围 |
| **结构** | 一段话 | 分段但混乱 | 明确的节标题或编号列表 |

### 快速自测问题

1. 如果换一个同事读我的 prompt，他知道要做什么吗？
2. 我知道什么算"完成"吗？（写得出来验收标准吗？）
3. 我说了"不要做什么"吗？
4. 我给了相关的文件路径吗？
5. 如果 AI 做错了，我知道错在哪里吗？

## 七、常见开发任务的提示词模板

### 7.1 功能开发

```
请在 [技术栈] 中实现 [功能]。
相关文件：@file:src/...

要求：
- 输入：[参数/数据来源]
- 输出：[返回值/副作用]
- 错误处理：[具体异常场景和处理方式]
- 风格：[遵循项目 CLAUDE.md 中的约定]

验证：
- [测试命令]
- [手动验证步骤]
```

### 7.2 Bug 修复

```
修复 [具体文件/函数] 中的 [bug 表现]。

复现：
1. [步骤]
2. [触发条件]
3. [实际结果 vs 期望结果]

错误信息（粘贴错误日志/堆栈，不超过 30 行）：
    [错误日志放这里，用 4 空格缩进]

约束：
- 不改变 [接口/行为]
- 已有测试继续通过

验证：运行 [测试命令] 确认修复
```

### 7.3 代码审查

```
请审查当前的 git diff。
重点关注：
- 正确性：逻辑错误、空指针、并发问题
- 安全性：注入风险、认证绕过、密钥泄露
- 边界条件：空输入、极端值、超时

对每个发现的问题给出：
- 严重度（严重/警告/建议）
- 文件:行号
- 修复建议（具体代码）
```

### 7.4 重构

```
重构 @file:src/old.py，目标：[提取公共逻辑 / 拆分为多个模块 / 改善性能]

约束：
- 不改变外部行为（已有测试必须全部通过）
- 不改变 public API 签名
- 遵循项目命名约定（见 CLAUDE.md）

验证：
1. 运行 [测试套件]
2. 运行 [lint 工具]
3. 对比重构前后的 [性能指标]
```

### 7.5 调试

```
分析以下错误：

错误：[错误类型和消息]
环境：[Python 版本 / 操作系统 / 关键依赖版本]
相关代码：@file:src/problem.py

出问题的代码（用 4 空格缩进粘贴，20-50 行）：
    def problem_function():
        ...

我期望：[描述正常行为]
实际发生：[描述异常行为]

请：
1. 分析可能的原因（按可能性排序）
2. 给出验证每个假设的方法（诊断命令）
3. 对于最可能的原因，给出修复方案
```

### 7.6 探索/理解代码

```
请分析 @file:src/module/ 的架构。

关注：
- 模块职责和边界
- 数据流（输入 → 处理 → 输出）
- 依赖关系（被谁调用、调用了谁）
- 潜在问题（循环依赖、过度耦合、缺少抽象）

输出格式：
1. 模块职责图（文本形式）
2. 关键数据流
3. 3-5 个设计风险（含缓解建议）
```

## 八、模型特定的提示词技巧

不同模型有不同的"脾气"。了解这些差异可以显著提升提示词效果。

> ⚠️ 以下建议基于 2025 年中各模型行为。模型版本迭代快，具体行为可能已变化。

| 模型 | 特定建议 |
|------|---------|
| **Claude Sonnet/Opus** | 将关键约束放在 prompt **末尾**（近因效应），同时开头简要提及；用 `<system-reminder>` 传递更新而非修改系统 prompt |
| **Claude Haiku** | 指令需更明确，少依赖隐含推理；适合"核查清单"式结构化任务 |
| **GPT-4o** | 对角色设定高度敏感；系统 prompt 中的角色定义效果强 |
| **GPT-o1/o3** | **讨厌** CoT 脚手架（"think step by step"）——它自己会推理，加入反而干扰 |
| **Gemini** | 指令放在数据**之后**（非之前）；对输出格式指令更敏感 |
| **DeepSeek** | 所有内容放用户 prompt（系统 prompt 效果差）；推理过程可能非常冗长 |
| **本地模型** | 需要更精确的格式示例；少用开放式指令，多给具体模板 |

## 九、渐进式澄清——从模糊到精准的对话流程

当使用者给出一个模糊请求时，按以下流程逐步澄清（每次只问 3-5 个问题）：

### 第 1 轮：目标澄清

```
我理解你想 [做 X]，还需要确认几个关键点：
1. 最终产出是什么？（一个文件？一个功能？一个修复？）
2. 这个任务的范围是？（只改一个函数？整个模块？多个服务？）
3. 有没有我绝对不能动的东西？
```

### 第 2 轮：约束澄清

```
明白了。在开始之前，关于约束：
1. 性能和可读性哪个优先？
2. 可以引入新的依赖吗？
3. 需要向后兼容吗？
```

### 第 3 轮：验证标准

```
清楚了。最后确认一下验收标准：
1. 什么测试/检查通过就算完成？
2. 需要更新文档吗？
3. 需要 commit 吗？commit message 格式？
```

### 退出条件

当以下所有条件满足时，不再追问，直接执行：
- ✅ 目标可以一句话说清楚
- ✅ 输入输出明确
- ✅ 约束无歧义
- ✅ 验证标准可执行

### 九.5、建议卡模式（v0.5 新增）

**在追问之前先给使用者建议**，而非让使用者自己摸索。核心原则：
1. 列出假设 → 2. 给 2-3 个可行方向 + 明确推荐 → 3. 追问带选项 → 4. 重写后给 A/B/C 判断权

此模式来自 `spec-driven-development` skill 的"假设先行"方法论。完整流程和触发条件见 `/clarify` 命令和 GLOBAL-CLAUDE.md §七。

触发词见 GLOBAL-CLAUDE.md §七（6 个简略触发词，自动启动 `/clarify` 建议卡模式）。

## 十一、速查卡片

### 发送 prompt 前的 30 秒检查

```
□ 目标：一句话能说清楚要达成什么吗？
□ 位置：给了文件路径或 @file 引用吗？
□ 约束：说了"不要做什么"吗？
□ 验证：写了"怎么证明完成"吗？
□ 结构：用了编号/标题/分段吗？
```

### 如果 AI 输出不满意

```
□ 先问自己：我是在给指令（怎么做）还是在给目标（做什么）？
□ 添加验证标准，让 AI 自己检查输出
□ 给 1 个简短示例做风格锚点
□ 把模糊词（"优化"/"改进"）替换为具体动作
□ 用 Echo Prompt 让 AI 帮你重写提示词
```

## 十三、快速自测：我属于哪种"写不好"？

回答以下 5 题，找到自己的典型弱点：

| # | 问题 | A（1分） | B（3分） | C（5分） |
|---|------|---------|---------|---------|
| 1 | 发 prompt 时会写文件路径吗？ | 从来不会 | 偶尔写模块名 | 都用 @file 或完整路径 |
| 2 | 会写"怎么验证完成"吗？ | 从来不会 | 会说"跑一下测试" | 会写具体测试命令 |
| 3 | 你的 prompt 多长？ | < 20 字一句话 | 一段话 | 分段/编号结构清晰 |
| 4 | 会写"不要做什么"吗？ | 从来不会 | 偶尔 | 每次都写排除范围 |
| 5 | prompt 是说目的还是说步骤？ | 说步骤（改哪行） | 都有 | 说目标（要什么结果） |

**得分**：
- **5-10 分** — "无帮手型"：最大问题是缺上下文和验证。优先养成"发 prompt 前补文件路径+验证命令"的习惯。
- **11-18 分** — "结构松散型"：知道要素但组织不好。优先用三要素框架重新组织每一条 prompt。
- **19-25 分** — "接近专业型"：基础扎实。可以用 CO-STAR 框架和 Echo Prompt 进一步精细控制。

---

> **记住**：提示词质量不是天赋，是可训练的肌肉。用好框架、清单和自动化，任何人都能从"说不清楚"跨越到"精准驱动"。
