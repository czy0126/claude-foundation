# 验证模式目录

> 从官方文档、社区实践、生产级验证系统中提炼的可复用模式。
> 核心原则：**不信任最后一步的输出——从不同角度独立验证，再提交。**

---

## 模式 1：运行完整性审计链

**来源**：生产级数据流水线完整性审计系统 + 官方 Workflow 模式

**适用场景**：任何多步骤自动化流水线（数据处理、模型训练、报告生成）

**模式**：
```python
def audit_run_integrity(*artifacts):
    """每次运行后，检查跨产物一致性"""
    issues = []
    for name, path, expected_date in artifacts:
        data = load_json(path)
        if data is None:
            issues.append(f"MISSING: {name}")
        elif data["as_of_date"] != expected_date:
            issues.append(f"DATE_MISMATCH: {name} (expected {expected_date})")
    return {"healthy": len(issues) == 0, "issues": issues}
```

**关键设计**：
- 检查**所有**预期输出文件存在（不只是"流水线跑完了"）
- 检查**日期一致性**（不能混入上一次运行的过期残留）
- 检查**文件修改时间戳**证明新鲜度（输出必须比输入新）
- **运行多次**：流水线中段跑一次 + 产出可视化后跑一次 + 关闭证据循环时跑一次
- 决策必须是结构化的：`HEALTHY` / `INCOMPLETE` / `FAILED`

**Claude Code 应用**：每当你让 Claude Code 生成多个文件，最后用一个检查步骤验证所有文件存在且一致。

## 模式 2：预检守卫

**来源**：生产级硬件资源守卫 + 官方 Hook 模式

**适用场景**：任何有前置条件的操作

**模式**：
```
操作前 → 收集前置条件状态 → 评估布尔门禁 → 通过则继续，失败则阻塞并报告
```

**实例**：
- 硬件守卫：CPU < 阈值？内存 < 阈值？GPU 显存 < 阈值？任一不满足 → block
- 数据守卫：输入文件存在？日期正确？行数 > 最小阈值？→ gate
- 环境守卫：Python 版本正确？依赖已安装？API key 已配置？→ gate

**Claude Code 应用**：
- 用 `PreToolUse` 钩子实现命令级守卫（如阻止 `rm -rf`）
- 用 `SessionStart` 钩子注入环境状态检查
- 在 CLAUDE.md 中列出硬性前置条件

## 模式 3：多步发送-验证循环

**来源**：7 步发送-验证-回退系统（消息发送 / API 调用 / 数据库写入等副作用操作）

**适用场景**：任何有副作用的操作（API 调用、文件写入、数据库变更、UI 操作）

**模式**：
```
1. 确认前置状态（截图/查询当前状态）
2. 执行操作
3. 截图/查询新状态 → 验证操作效果
4. 如果未达预期 → 尝试替代路径
5. 再次截图/查询 → 最终确认
6. 返回 bool（不是 void！）
```

**关键设计**：
- 操作成功 ≠ 效果达成（API 返回 200 ≠ 数据真正写入了）
- 每次验证都必须是**独立来源**的新鲜截图/查询
- 函数返回 `bool`——调用方必须检查返回值

**Claude Code 应用**：写完文件后用 `Read` 或 `Bash(cat ...)` 回去验证内容、运行测试验证行为。

## 模式 4：对抗性交叉验证

**来源**：跨模型验证门禁系统（Jaccard 一致性 / 逻辑等价检查）+ 官方 Workflow 对抗验证 + 学术研究 (CriticGPT, NeurIPS 2024; Reflexion, 2023)

**适用场景**：任何高风险决策或 AI 生成内容的验证

**学术证据**：跨模型审查比同模型自审多发现 **15-25%** 的问题；LLM 自审遗漏自身 bug 的概率达 **68%**；被指出 bug 后 **23%** 的情况下会编造合理解释而非承认错误。

**模式**：
```
模型 A 输出 ←→ 模型 B 输出（独立，不同角度）
         ↓
    一致性检查（Jaccard / 逻辑等价 / 结构比对）
         ↓
    ┌─ 一致 → 通过
    └─ 不一致 → 标记人工审核 / 阻塞
```

**变体**：

| 变体 | 有效性 | 方法 | 适用场景 |
|------|--------|------|---------|
| **测试执行反馈** | ⭐⭐⭐⭐⭐ 捕获 60-80% 幻觉 | 先写测试→验证失败→实现→验证通过 | **所有场景（黄金标准）** |
| **双模型审查** | ⭐⭐⭐⭐ 多发现 15-25% | Sonnet 生成 → Haiku/Opus 独立审查 | 架构决策、关键逻辑 |
| **自我对抗** | ⭐⭐⭐ | 实现后切换角色："以审查者身份找出 3 个问题" | 日常代码审查 |
| **双角度** | ⭐⭐⭐ | 同一模型，不同 prompt（正确性 + 安全性） | 安全敏感代码 |
| **独立来源交叉** | ⭐⭐⭐ | 不与模型输出交叉，与源码/测试/git 历史交叉 | 所有场景 |

**关键警示**：更大模型更"谄媚"（更容易附和用户错误），因此用**较小模型审查较大模型**（如 Haiku → Sonnet）可能比同一大模型自审更有效。

**Claude Code 应用**：
- 关键变更后："/code-review 审查我刚才的变更"
- 复杂逻辑：用不同 prompt 让 Claude 从不同角度分析同一段代码
- 检查清单：定义硬性通过标准，不让模型自我评分
- 双模型：如果可用，让 Haiku 快速审查 Sonnet 的关键输出

## 模式 5：新鲜度检查

**来源**：数据新鲜度检查（文件修改时间戳比对）+ 官方文档

**适用场景**：任何有"数据新鲜度"要求的自动化流程

**模式**：
```python
if monitor_mtime < history_mtime:
    issues.append("stale_monitor")  # 监控图比数据源还旧 → 过期！
```

**检查项**：
- 输出文件的修改时间 > 输入文件的修改时间
- 缓存文件的修改时间 > 上次数据刷新时间
- 报告中的日期字段 == 预期的运行日期（不是墙钟日期）

**Claude Code 应用**：生成代码后运行测试确认测试文件比源文件新；压缩后用 `PostCompact` 钩子重新注入关键上下文。

## 模式 6：故障自愈与崩溃恢复

**来源**：分布式计算容错引擎（检查点/重试/降级）+ 官方 Worktree 模式

**模式**：
```
操作
 ├─ 成功 → 保存检查点 → 继续
 └─ 失败 → 重试（3次，降级资源）
      ├─ 成功 → 保存检查点 → 继续
      └─ 全部失败 → 标记失败 → 跳过（不要阻塞整体流程）
           └─ 下一轮：resume_failed() 重新尝试
```

**关键设计**：
- 检查点保存频率 = 每次迭代（最多丢失 1 次迭代的工作）
- `auto_clean()` 在拟合前删除失败/异常记录（防止污染模型）
- 异常检测：`is_valid()` 过滤超出物理范围的数据（Tmax ∉ [20,200]°C）

**Claude Code 应用**：
- 使用 git worktree 进行隔离（失败不污染主工作区）
- 长时间会话定期 checkpoint（"/summarize 当前进展，写入 CLAUDE.md"）
- 两次修正失败 → 停止，重新分析根因

## 模式 7：持续漂移监控

**来源**：模型衰减监控系统 + 官方模型最佳实践

**适用场景**：任何随时间衰减的系统（ML 模型、缓存、配置）

**模式**：
```
不是"训练一次就信任"，而是持续监控：
├─ 滚动窗口表现 vs 基准
├─ 近期回撤和连续下行天数
├─ 分数 PSI（群体稳定性指数）—— 组件级漂移
├─ 验证分数质量（rank IC、top-bucket 命中率）
└─ 重训练门禁：要求 N 个有效观察日 → 不要因为一天坏就追参数
```

**Claude Code 应用**：CLAUDE.md 中的"已知陷阱"标注日期 → 过期的陷阱可能已不再适用，但不要删除（有日期可以判断）。

## 验证检查清单（每个任务结束时）

- [ ] 所有预期输出文件存在且日期一致
- [ ] 测试通过（不仅是"跑过了"，要确认全部绿色）
- [ ] git diff 逐行审查无意外变更
- [ ] 如果是 UI 变更：实际运行并目视验证
- [ ] 如果是 API 变更：用 curl 验证
- [ ] 对关键逻辑：运行对抗验证（自我审查或交叉模型）
- [ ] 更新 CLAUDE.md 中的当前状态
